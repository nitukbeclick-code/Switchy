# Architecture

חוסך / **Switch AI (Switchy)** is an Israeli telecom price-comparison and
lead-generation product. It is Hebrew-first and RTL throughout. The repository
holds **four surfaces** that share one catalogue of real plans and one Supabase
backend.

> Honesty / E-E-A-T is a hard rule across every surface: no fabricated plans,
> prices, ratings, reviews, or compliance claims. Every comparison renders real
> catalogue rows only.

## The four surfaces

| # | Surface | Path | Stack | Status |
|---|---------|------|-------|--------|
| 1 | Flutter app | `lib/` | Flutter (hand-authored, no FlutterFlow); `go_router`, `provider`, `google_fonts`, `flutter_animate` | Mobile + web target |
| 2 | Next.js GEO web app | `web/` | Next.js 16 (App Router), React 19, Tailwind v4, TypeScript | **Live** at `https://switchyy-omega.vercel.app` |
| 3 | Static marketing site | `site/` | Hand-tuned HTML generated by `site/build.js` (Node, no deps) | **Live** at `switchy-ai.com` |
| 4 | Supabase backend | `supabase/` | Postgres + RLS + `pg_cron` + Deno Edge Functions | Live project `orzitfqmlvopujsoyigr` |

The two web front-ends (`web/` and `site/`) coexist on purpose: `site/` is the
older static SEO site that is still the live `switchy-ai.com`, and `web/` is the
newer Next.js "GEO" (generative-engine-optimization) app that is live on Vercel.
Both link into the same Supabase lead pipeline.

## 1. Flutter app (`lib/`)

Plain, hand-authored Flutter — **no FlutterFlow** (see the architecture
directive in `CLAUDE.md`). Highlights:

- **Foundation** — design tokens in `lib/theme/app_theme.dart`
  (`AppTheme.of(context)`, `AppColors.*`); navigation in `lib/core/nav.dart`
  (re-exports `go_router`, adds `context.safePop()`); the primary CTA in
  `lib/widgets/app_button.dart` (`AppButton`, async loading state); app state in
  `lib/app_state.dart` (a `ChangeNotifier` singleton over `SharedPreferences`,
  exposed via `Provider`).
- **Services** — pure, unit-tested domain logic lives in `lib/services/` and is
  the single source of truth for each concern (recommendation engine,
  notifications, provider ratings, renewal report, search, savings summary,
  media). Screens render service output; they do not re-derive it.
- **Backend seam** — `lib/services/backend/` has a `Backend` interface with two
  implementations: `LocalBackend` (the on-device default, no keys) and
  `SupabaseBackend` (live only when build-time Supabase defines are present). So
  `flutter test` / CI run fully on-device with no project.
- **Catalogue** — `lib/data.dart` + `lib/data/plans_*.dart` hold the static plan
  catalogue; `lib/models.dart` the types. This is the upstream source of the
  catalogue the web surfaces consume (exported to `site/data/plans.json`).
- **Pages** — one file per page under `lib/pages/<name>/<name>_widget.dart`.

## 2. Next.js GEO web app (`web/`)

App-Router Next.js 16 (React 19, Tailwind v4, TypeScript). It is a
content + lead-funnel surface tuned for both classic SEO and generative-engine
answers ("GEO"). Key facts:

- **The build needs no secrets.** The catalogue is bundled at
  `web/data/catalogue.json` (a copy of `site/data/plans.json`) and read once at
  module init by `web/lib/data.ts`. Providers are *derived* from the plans
  (unique names → slug + stats). All catalogue exports are synchronous and
  side-effect-free so pages can call them in RSC render / `generateStaticParams`.
- **Routes** (`web/app/`): `/` (home), `/compare/[service]`, `/providers/[slug]`,
  `/switch/[provider]`, `/glossary` + `/glossary/[term]`, `/market-pulse`,
  `/transparency`, plus `sitemap.ts`, `robots.ts`, and an LLM-context surface
  (`/api/llm-feed`, `/api/llm-feed.json`, `app/llm-context.txt`).
- **Content sources**: `web/data/{catalogue,glossary,cities}.json`, FAQ in
  `web/lib/faq.ts`, JSON-LD schema builders in `web/lib/schema.ts`.
- **Legacy slug redirects**: `web/next.config.ts` 301-redirects old opaque
  provider slugs (`/providers/p-*`, `/switch/p-*`) to the new readable ones so
  indexed URLs never 404.
- **⚠️ Next 16 caveat**: this is not the Next.js in your training data. Read
  `web/node_modules/next/dist/docs/` before writing Next-specific code
  (`web/AGENTS.md`).

### Lead funnel (web)

```
web/components/LeadForm.tsx
        │ POST application/json
        ▼
web/app/api/lead/route.ts   (runtime: nodejs, force-dynamic)
        │  • Origin allow-list (CSRF guard)
        │  • requires SUPABASE_SERVICE_ROLE_KEY (server-only) → 503 if absent
        │  • mandatory terms + privacy consent → 400 if false
        │  • Israeli phone normalization (web/lib/phone.ts, shared with the form)
        ▼
INSERT public.leads   (service-role; DB rate-limit + consent-stamp triggers apply)
        ▼
notify-lead trigger  →  Telegram / Resend / AI triage   (see DATA_MODEL + EDGE_FUNCTIONS)
```

Conversion tracking (`web/lib/tracking.ts`) fires GA4 + optional Meta Pixel on a
*confirmed* lead-submit success only — never on page view. Both no-op safely
when the analytics globals or the pixel id are absent.

## 3. Static marketing site (`site/`)

Hand-tuned static HTML, regenerated by `node site/build.js` (zero npm deps). The
builder emits ~230 category / provider / compare / guide / 404 pages plus
`sitemap.xml` and `robots.txt` from templates + the catalogue exported to
`site/data/plans.json`, and content fragments under `site/content/guides/`
(~142 guide JSON files). `index.html` is hand-written, not generated. CSS/JS are
content-fingerprinted (`?v=<hash>`) for cache-busting under the immutable
headers in `vercel.json`. The site's lead form (`script.js` → `sendLead()`)
posts to the same `public.leads` table via the public anon key.

## 4. Supabase backend (`supabase/`)

One Postgres project (`orzitfqmlvopujsoyigr`, region `eu-central-1`). It holds
the schema + RLS (`schema.sql` and dated migration `*.sql` files), `pg_cron`
schedules, and the Deno Edge Functions under `supabase/functions/` (shared code
in `_shared/`). The functions are the product's server-side brain:

- **Lead/CRM pipeline** — `notify-lead` (the interactive Telegram "digital rep"
  fired on every new lead **and** meeting), `renewal-reminders` (the scheduled
  digest / sweep / SLA-follow-up / weekly-report brain), `crm-api` (admin CRM
  backend for the WhatsApp pipeline).
- **Site AI** — `site-ai-chat`, `site-plan-advisor`, `site-bill-analyzer`,
  `site-subscribe` — public endpoints grounded strictly in a bundled catalogue
  snapshot.
- **Community** — `community-moderate` (LLM spam/abuse flagging on insert),
  `community-notify` (team Telegram ping on new community activity).
- **Channels** — `whatsapp-webhook` (Meta Cloud API AI agent + CRM),
  `telegram-webhook` (deep-link account linking), `analytics-track` (funnel
  event sink).

Full per-function detail (purpose, auth model, deploy recipe) is in
[`EDGE_FUNCTIONS.md`](./EDGE_FUNCTIONS.md). The data model + RLS posture is in
[`DATA_MODEL.md`](./DATA_MODEL.md).

## Cross-surface data flow summary

| Flow | Source | Sink | Mechanism |
|------|--------|------|-----------|
| Catalogue | `lib/data/` | `site/data/plans.json`, `web/data/catalogue.json` | export tool, copied/bundled at build |
| Lead | Flutter app / `web/` `/api/lead` / `site/` form | `public.leads` | RLS-gated INSERT (consent + rate-limit triggers) |
| Lead notify | `public.leads` INSERT | Telegram + Resend + AI | AFTER INSERT trigger → `pg_net` → `notify-lead` |
| Meeting | Flutter booking wizard | `public.meetings` → Telegram card | trigger → `notify-lead`; Zoom S2S OAuth or reply-with-link |
| WhatsApp CRM | Meta Cloud API | `whatsapp_*` + `crm_events` | `whatsapp-webhook` → service-role writes; `crm-api` for admin reads |
