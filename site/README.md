# site/ — switchy-ai.com (static SEO site)

The server-rendered marketing + SEO site for SWITCHY: plain HTML/CSS/JS, RTL
Hebrew, no framework. **Every page except `index.html` and `og-card.html` is
generated by `build.js`** — a single dependency-free Node script (no
`package.json`, nothing to install). The generated output is **committed to the
repo**, and Vercel serves the `site/` directory as-is via its git integration
(no build command on the host).

```bash
node site/build.js        # regenerate everything — this is also the validation gate
cd site && python3 -m http.server 8000   # preview at http://localhost:8000
```

A successful run ends with the page-count summary, the asset fingerprints, and
a provenance line saying whether the catalogue was read **LIVE from Supabase**
or from the **bundled snapshot**.

## Data: live catalogue with a never-blank fallback

- At build time the plan catalogue is read **live from Supabase
  `public.plans`** over the REST API using the *public* anon key (no secret
  anywhere in this path; 8s timeout, isolated child-process fetch).
- `data/plans.json` (the last-known-good snapshot) is **always loaded first**:
  it is the fallback on *any* live-read failure and the merge source for
  qualitative fields (perks / fine print) that live rows may lack. The build
  therefore cannot render blank or fabricated data — at worst it's one snapshot
  behind and self-heals on the next run.
- Which source won is stamped into `index.html` as
  `<meta name="build:catalogue-source" content="live|bundled">`.
- Truth-only rule: visible HTML, schema.org `Offer`/`AggregateOffer` JSON-LD and
  the `window.__PLANS__` blob are all rendered from the *same* in-memory
  catalogue, so they can never disagree. Every count/price/date in `llms.txt`,
  `ai.txt` and the sitemap derives from the real catalogue too.

## What gets generated

Roughly 280 `.html` pages (277 at the time of writing — the exact set tracks
the catalogue, since provider head-to-heads are partly auto-paired and
collections only publish with ≥3 matching plans):

- 5 category landings (`cellular` / `internet` / `tv` / `triple` / `abroad`)
- 18 collection pages (e.g. `cellular-5g`, `plans-no-commitment`, `kosher-plans`)
- 4 calculators (`calc-*.html`), 3 concept versus pages (e.g. `fiber-vs-cable`)
- ~69 provider head-to-heads (15 curated + auto pairs per category)
- 18 provider pages + `providers.html`
- 150 guides (`guide-*.html`: 8 authored inline in `build.js`, 142 as JSON files
  in `content/guides/`) + `guides.html`
- Hubs/conversion: `plans`, `deals`, `compare`, `comparisons`, `community`,
  `book`, `app`, `faq`, `glossary`, `how-it-works`
- 5 static/legal pages (`about`, `privacy`, `terms`, `account-deletion`,
  `accessibility`) and a noindex `404.html`
- Plus `sitemap.xml`, `robots.txt`, `llms.txt`, `ai.txt`

**Prune pass:** any top-level `.html` a run did *not* emit (except the two
hand-authored files) is deleted — stale pages can't linger. Consequences:
never hand-edit a generated page (the next rebuild overwrites it) and never
hand-create a new `.html` here (the next rebuild deletes it).

**Asset fingerprinting:** `styles.css`, `script.js` and `translate-runtime.js`
are referenced as `?v=<sha256-8>` content hashes; the deploy configs cache them
immutable for a year, so a changed file gets a new URL instantly.

**Hand-authored files:** `index.html` (the homepage) and `og-card.html` (the
source rendered to `og-image.png`). The build syncs `index.html` *in place* —
the `?v=` hashes, the `window.__HERO_PLANS__` blob, the deal-ticker items and
the provenance meta — but never touches its copy. Its header/footer/mega-menu
are maintained by hand and must be kept in step with `navHtml()` in `build.js`.

## Rebuild + deploy loop

`.github/workflows/rebuild-static.yml` runs `node site/build.js` on:

1. a **30-minute cron** (near-real-time price refresh, no human in the loop),
2. manual `workflow_dispatch`, and
3. `repository_dispatch` (`catalogue-updated`) fired by a Supabase database
   webhook — an instant rebuild the moment a plan row changes.

It commits **only the build outputs** (`*.html`, `sitemap.xml`, `robots.txt`,
`llms.txt`, `ai.txt`) when they actually changed; that push makes Vercel
redeploy. Net effect: editing prices in the Supabase dashboard reaches
production with no manual git push. Identical content produces byte-identical
output (evergreen sitemap `lastmod`s are pinned, not build-dated), so quiet
days produce no commits.

## Editing

- **Copy, data, templates:** edit `build.js`, then rerun it. If you change
  evergreen page content (about/legal/app/book/community/glossary), bump
  `EVERGREEN_LASTMOD` in the sitemap block.
- **New guide:** drop a JSON file into `content/guides/` (copy an existing one
  for the shape) — no generator change needed; it's picked up by the guides
  index, mega-menu, FAQ hub and sitemap automatically.
- **Homepage:** edit `index.html` directly, then rebuild so the synced bits
  refresh.
- **Client behaviour:** `script.js` (site-wide features) and
  `translate-runtime.js` (16-language client translation, shipped byte-identical
  to the Next app).
- **Guardrails:** prices/savings figures come only from the catalogue — never
  hardcode them; the legal pages' copy (privacy/terms/accessibility/
  account-deletion) changes only deliberately; per-provider brand colors are
  never remapped to the site palette.

## Deploy configs

`vercel.json` is the live one (security headers + 1-year immutable caching for
fingerprinted assets). `netlify.toml` and `_headers` are parallel configs for
alternate hosts — if you touch header policy, copy `vercel.json`'s values
verbatim so they don't drift.

## Relationship to the rest of the repo

This directory is the **desktop/SEO front door**. The Next.js app in `web/`
serves phones (its middleware rewrites desktop user agents to this static
site), and the Flutter app is the product. Brand tokens here mirror
`lib/theme/app_theme.dart`; analytics event names follow the shared taxonomy in
`docs/events.md`.
